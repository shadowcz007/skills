# Usecase Generator

根据输入的 URL 或原始信息，自动抓取内容并整理成标准格式的 usecase markdown 文件。

## 工作流程

### 第一步：解析输入

识别用户提供的所有来源：

- **URL 列表**：识别 `[数字] 标题: URL` 格式，提取编号、标题、URL
- **纯文本信息**：如果用户提供的是描述性文字而非 URL，跳过抓取步骤，直接进入整理步骤
- **混合格式**：部分有 URL、部分只有描述

### 第二步：获取正文（仅针对 URL）

按以下优先级尝试获取内容：

1. **Playwright MCP**（首选）：使用 `browser_navigate` + `browser_snapshot` 获取页面内容
2. **WebFetch**（备选）：如果 Playwright 不可用或失败，尝试 WebFetch
3. **标注失败**：如果某个 URL 完全无法获取，标注出来，继续处理其他 URL

**注意**：ScienceDirect (doi.org)、Google Scholar 等有验证码墙的网站可能抓取失败，这是正常现象——标注失败并继续。

### 第三步：内容理解

阅读获取到的正文内容，理解以下要素：

- **人物**：这是关于谁？什么背景？有什么成就？
- **场景**：他们遇到了什么问题或需求？
- **方法**：他们用什么工具、方法、流程解决了问题？
- **成果**：产生了什么？影响了谁？

### 第四步：按模板整理

每个 URL 对应一个独立的 usecase 文件。使用以下格式：

```markdown
# 案例：[简洁描述这是关于什么的]

> 来源：[URL]

## 一句话说清楚
用 1-2 句话概括这个案例的核心价值，让读者 5 秒内判断是否值得继续看。

## 适合谁 / 不适合谁
**适合：** 具体目标人群
**不太适合：** 不适合的人群

## 为什么重要
这个案例对读者有什么启发？为什么值得了解？

## 人物
描述人物背景、头衔、成就。如果是多人案例，分别描述。

## 场景
他们面临什么具体问题或需求？上下文是什么？

## 方法
系统如何运作？核心设计是什么？用流程叙事或清晰的结构化描述，避免简单罗列。

## 几个具体设计值得借鉴（可选）
如果方法中有特别值得借鉴的细节，抽出来单独说明。

## 影响力 / 成果
产生了什么具体成果？影响了多少人？量化数据。

## 阅读难度
★☆☆☆☆ ~ ★★★★★（1-5 颗星）
实操难度：★☆☆☆☆ ~ ★★★★★（1-5 颗星）

## 可以带走什么
给读者的 2-3 个立即可行动的建议或启发。
```

### 第五步：保存文件

- 文件保存到当前目录或用户指定目录
- 文件名格式：`[序号]-[简短 slug].md`
- 例如：`01-pedro-santanna-claude-code-workflow.md`

## 输出要求

1. 每个 URL 生成一个独立的 `.md` 文件
2. 文件内包含来源 URL
3. 严格按照模板格式，确保包含「一说」「适合谁」「为什么重要」「可以带走什么」
4. 术语需加通俗语境解释（目标读者是普通用户，不是专家）
5. 方法部分用叙事逻辑，避免简单 bullet 罗列

## 特殊情况处理

- **无 URL 仅有描述**：直接进入整理步骤，按上述模板填充能推断的内容，其他标注「信息不足」
- **部分 URL 抓取失败**：成功抓取的整理成 usecase，失败的单独标注「无法抓取，建议手动访问」
- **长 URL 列表**：逐个处理，每完成一个告知用户进度
